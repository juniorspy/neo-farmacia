# Session 2026-04-11-01: Multi-tenant provisioning end-to-end

## What was done

Single long session that turned a working single-tenant deployment into a functional multi-tenant platform. Six stages of work, all merged and deployed to `api.leofarmacia.com` + `app.leofarmacia.com`.

### Stage-by-stage

**Phase A — Multi-tenant backend foundation** (commits `257ab7f`, `46e7d9d`)

1. `seedDefaultStore()` — adopts the existing `odoo` DB as `store_leo` / Farmacia Leo on API startup. Idempotent. First tenant keeps all 110 existing products.
2. `DELETE /api/v1/admin/pharmacies/:id?confirm=yes` — reverse-order cleanup (Meilisearch → Odoo DB → Mongo records). Refuses default store. Requires explicit confirm query param.
3. Random per-pharmacy admin passwords — 24-char base64url, bcrypt hash on Store doc, plaintext carried through job step data only to the steps that need it and the email step for later retrieval.
4. Partial-failure verification — `odoo_db_create` step authenticates as the expected admin to verify the DB is actually healthy (not a half-created corpse from a previous crash). Drops and recreates if unhealthy. Post-create verification fails loud if master password is wrong.
5. **`request.store` middleware + scoped Odoo routing** — the big one. New `store-context.plugin.ts`, `odoo-scoped-cache.ts`, `odoo-store-ops.ts`. Refactored `orders`, `products`, `stats` to use `request.odoo` instead of the global singleton. Added `request.store` + `request.odoo` types to Fastify.
6. Super-admin UI at `/admin/pharmacies` — list table with status badges, auto-refresh every 3s while provisioning, create modal, details drawer with step-by-step job progress, retry + delete buttons.

**Phase A.5 — Closing the onboarding gap** (commit `6ac8d96`)

1. New 7th step `create_dashboard_admin` — creates a pharmacist-role `Admin` doc in Mongo with `owner_email` as login and the same random password as Odoo. Idempotent: appends store to existing admin if email already exists.
2. Credentials visible in admin UI — `email_credentials` step now retains plaintext password in step data until super-admin clicks "Marcar como entregadas". New endpoint `POST /admin/pharmacies/:id/credentials/mark-delivered` scrubs. Details drawer shows an amber panel with copy-to-clipboard for URL/email/password.
3. Super-admin store switcher — `Shell.tsx` fetches `/admin/pharmacies` for role=admin users and feeds active pharmacies into the existing header dropdown. Lets super-admin view any tenant's data.

**Phase B1–B5 — Make the WhatsApp→Odoo loop multi-tenant** (commit `404fc44`)

1. **B1**: `catalog-sync.service` iterates all active stores on each tick. Replaced hardcoded `['store_leo']`. Each new pharmacy gets its Meilisearch index populated automatically.
2. **B2**: `CommandContext` gains `store` + `odoo` fields. `commands.routes` resolves the store and attaches a scoped client before dispatching. `pedido.handler` rewritten to use `ctx.odoo` for every RPC.
3. **B3**: New `store-resolver.ts` — resolves Store by `whatsapp_instance_id` with a 60s in-process cache. Webhook now drops messages from unknown instances instead of silently misrouting them.
4. **B4**: Webhook forwards `store_config { name, currency, timezone, lang, agent: { name, greeting_style, signature, business_hours, delivery_info, custom_notes } }` to n8n alongside the message text. n8n agents read this to personalize replies.
5. **B5**: New `/agent` "Mi Agente" page in the dashboard with form + live greeting preview. New `GET /api/v1/stores/:storeId` and `PATCH /api/v1/stores/:storeId/agent-config` endpoints (scoped, length-validated, cache-invalidating).

### ADRs + docs written

- **ADR-007 — Tiered POS sync-back strategy** (`docs/decisions/007-tiered-pos-sync.md`): <5 units human-in-the-loop, 5-10 real-time push, >10 nightly reconciliation. Solves concurrent overselling naturally at the critical tier via serialized pharmacist confirmations.
- **Risk #11 in `known-risks.md`** — walk-in oversell while WhatsApp sells low-stock items.
- **Stage 6 (POS Sync)** updated with the bidirectional flow, expanded `PosAdapter` interface, Option A/B onboarding modes.
- **Stage 8 (new) — Multi-tenant Provisioning** — full architecture doc for the work done this session.
- **`testing-pending.md`** — explicit list of untested flows, sorted by criticality, with a suggested single smoke-test path that covers 80% of the red rows.

### Infrastructure changes

- **Odoo master password** set via a shell entrypoint wrapper in `docker-compose.yml` that writes `admin_passwd = ${ODOO_ADMIN_PASSWD}` into `/etc/odoo/odoo.conf` before exec-ing Odoo. Odoo 17 has no `--admin-passwd` CLI flag — required this approach.
- **`pos` service migrated from raw-compose-in-Dokploy to GitHub-backed** so the compose file in the repo is actually the source of truth. Previously, editing the repo had no effect on `pos`.
- **Dashboard domain** was accidentally set to `localhost` in the Dokploy Domain tab after some earlier edit — fixed to `app.leofarmacia.com`, traffic restored.
- **Memory saved**: repo is source of truth, secrets in Dokploy env, never hardcode on VPS. `pos` exception noted (raw → github migration done this session).
